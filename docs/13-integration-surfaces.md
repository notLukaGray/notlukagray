## Canonical Contract Surface

**Package:** `@pb/contracts`

The canonical contract is the interoperability boundary. Any tool that produces or
consumes page-builder JSON must validate against it.

### What it must contain

1. Zod schemas for every top-level type:
   - `pageBuilderSchema`
   - `sectionSchema` (and all section type variants)
   - `elementSchema` (and all element type variants)
   - `moduleSchema`, `modalSchema`, `globalsSchema`

2. Generated JSON Schema outputs:
   - One `.json` file per top-level type
   - Stored in `packages/contracts/dist/schemas/`
   - Regenerated by `npm run generate-schemas` in that package
   - Committed — consumers can use them without a build step

3. Contract version metadata:
   - `CONTRACT_VERSION` — current version string (`"1.0.0"` at first extraction)
   - `SUPPORTED_CONTRACT_VERSIONS` — array of versions this package can validate
   - Version embedded in the generated JSON Schema `$id` field

4. Capability contract schemas (for integrators):
   - `importerCapabilitySchema` — fields: `name`, `version`, `supportedContractVersions`,
     `supportedElementTypes`, `supportedSectionTypes`, `diagnosticCodes`
   - `exporterCapabilitySchema` — fields: `name`, `version`, `inputContractVersions`,
     `outputTargets`, `fidelityLevel`, `diagnosticCodes`
   - `cmsAdapterCapabilitySchema` — fields: `name`, `version`, `cmsProvider`, `syncModes`,
     `mappingFeatures`, `conflictPolicies`, `diagnosticCodes`

### Verification

- [x] `import { pageBuilderSchema } from "@pb/contracts"` works in a plain Node.js script
      (no Next.js, no framework)
- [x] JSON Schema files exist in `dist/schemas/` and are valid JSON Schema (draft-2020-12)
- [x] `CONTRACT_VERSION` is exported and equals the version in `package.json`
- [x] All 5+ Gate D fixtures validate against `pageBuilderSchema` with zero errors
- [x] Capability schemas validate example objects (one per type, committed as fixtures)

## Runtime API Surface

**Package:** `@pb/core`

### Functions

Each function has a defined input/output contract. Deviations from these signatures
are breaking changes.

```ts
// Load raw JSON from a file path or URL string
loadPage(source: string): Promise<unknown>

// Expand presets, modules, and defaults into canonical form
expandPage(raw: unknown): Promise<PageBuilderPage>

// Resolve asset references to final URLs (server-only)
resolveAssets(page: PageBuilderPage, options: AssetResolveOptions): Promise<PageBuilderPage>

// Validate against contract schema — returns diagnostics, does not throw
validatePage(page: unknown): ValidationResult

// Migrate from one contract version to another
migratePage(page: unknown, from: string, to: string): MigrationResult
```

### `ValidationResult` shape

```ts
type ValidationResult = {
  valid: boolean;
  diagnostics: PageBuilderDiagnostic[];
};

type PageBuilderDiagnostic = {
  code: string; // machine-readable, stable across versions
  severity: "error" | "warning" | "info";
  path: string; // JSON path to the offending field
  message: string; // human-readable
  contractVersion: string;
};
```

### `MigrationResult` shape

```ts
type MigrationResult = {
  page: unknown; // migrated page JSON
  diagnostics: PageBuilderDiagnostic[];
  fromVersion: string;
  toVersion: string;
  appliedTransforms: string[]; // names of transforms applied, in order
};
```

### `CoreConfig` — host configuration interface

This replaces the old static `@/app/theme` import. App wires this at startup.

```ts
type CoreConfig = {
  builderDefaults?: PbBuilderDefaults;
  contentGuidelines?: PbContentGuidelines;
  assetBaseUrl?: string;
  defaultSection?: Partial<SectionBlock>;
  defaultElement?: Record<string, unknown>;
};
```

### Verification

- [x] All five pipeline functions are exported from `@pb/core`
- [x] `validatePage` on a valid fixture returns `{ valid: true, diagnostics: [] }`
- [x] `validatePage` on a fixture with a known bad field returns a diagnostic with a
      stable `code` string
- [x] `migratePage` with an identity migration (same version in and out) returns the
      input unchanged
- [x] `migratePage` with a v0→v1 path returns a valid v1 page (fixture test)
- [x] None of the five functions import from `next/`, `react`, or any `@/app` path
- [x] App-level defaults are wired via `setCoreConfig` (no app usage of `setPageBuilderHostConfig`)

## CLI Surface

**Package:** `tools/pb-cli` (extracted from `scripts/pb-cli.ts`)

Gate D produced a working stub. This section specifies the full implementation.

### Commands

#### `pb-cli validate <file>`

```
pb-cli validate src/content/pages/work/algolia/page.json
```

Output (stdout, JSON):

```json
{
  "file": "src/content/pages/work/algolia/page.json",
  "valid": true,
  "contractVersion": "1.0.0",
  "diagnostics": []
}
```

Exit code 0 on valid. Exit code 1 on invalid. Exit code 2 on file-not-found or
parse error.

#### `pb-cli diff <file-a> <file-b>`

```
pb-cli diff page-v1.json page-v2.json
```

Output (stdout, JSON):

```json
{
  "contractVersion": "1.0.0",
  "changes": [
    {
      "path": "sections[0].elements[2].type",
      "from": "heading",
      "to": "body",
      "breaking": false
    }
  ]
}
```

#### `pb-cli migrate <file> --from <version> --to <version>`

```
pb-cli migrate old-page.json --from 0.9 --to 1.0.0
```

Output: migrated JSON written to stdout (pipe to file) plus a diagnostics summary
on stderr. Exit code 0 on success. Exit code 1 if migration path does not exist.

#### `pb-cli conformance <fixtures-dir>`

```
pb-cli conformance packages/contracts/fixtures/
```

Runs all `.json` files in the directory through `validatePage`. Output (stdout, JSON):

```json
{
  "contractVersion": "1.0.0",
  "total": 5,
  "passed": 5,
  "failed": 0,
  "results": [{ "file": "algolia.json", "passed": true, "diagnostics": [] }]
}
```

### Verification

- [x] `pb-cli validate` exits 0 on a known-valid fixture
- [x] `pb-cli validate` exits 1 on a fixture with a deliberately injected bad field
- [x] `pb-cli diff` output is valid JSON matching the schema above
- [x] `pb-cli migrate` outputs valid JSON for a v0→v1 migration
- [x] `pb-cli migrate` exits 1 and emits a structured error when path doesn't exist
- [x] `pb-cli conformance` runs all fixtures and all pass (`6/6` as of April 8, 2026)
- [x] All commands work without a Next.js server running

## SDK Surface

**Package:** `@pb/sdk`

The SDK wraps `@pb/core` with an ergonomic API for external integrators who want
programmatic access without the CLI.

### Client interface

```ts
import { createPbClient } from "@pb/sdk";

const pb = createPbClient({
  contractVersion: "1.0.0",
  // optional overrides
});

// Validate a page object
const result = await pb.validate(pageJson);
// result.valid, result.diagnostics

// Diff two pages
const diff = await pb.diff(pageJsonA, pageJsonB);
// diff.changes[]

// Migrate a page
const migrated = await pb.migrate(pageJson, { to: "1.0.0" });
// migrated.page, migrated.diagnostics

// Load from file path (Node.js only)
const page = await pb.load("./content/pages/work/algolia/page.json");
```

### Verification

- [x] `createPbClient` works in a plain Node.js script
- [x] All four methods are implemented; `validate/diff/migrate` are exercised via extracted CLI
- [x] `diagnostics` output matches `PageBuilderDiagnostic` shape from `@pb/core`
- [x] TypeScript types are exported and accurate (no `any` in public API)

## Plugin Interface Surface

**Package:** `@pb/extensions`

### Importer plugin

An importer takes foreign content and produces valid page-builder JSON.

```ts
type ImporterPlugin = {
  name: string;
  version: string;
  supportedContractVersions: string[];
  supportedElementTypes: string[];
  supportedSectionTypes: string[];
  diagnosticCodes: Record<string, string>; // code → description
  import(source: unknown): Promise<ImportResult>;
};

type ImportResult = {
  pages: unknown[];
  diagnostics: PageBuilderDiagnostic[];
  unsupported: UnsupportedConstruct[];
};

type UnsupportedConstruct = {
  code: string;
  path: string;
  description: string;
};
```

### Exporter plugin

An exporter takes a resolved page-builder page and produces output in a target format.

```ts
type ExporterPlugin = {
  name: string;
  version: string;
  inputContractVersions: string[];
  outputTargets: string[];
  fidelityLevel: "full" | "lossy" | "structural";
  diagnosticCodes: Record<string, string>;
  export(page: PageBuilderPage): Promise<ExportResult>;
};

type ExportResult = {
  output: unknown;
  diagnostics: PageBuilderDiagnostic[];
  droppedFields: string[];
};
```

### CMS adapter plugin

A CMS adapter syncs between a CMS source model and canonical page-builder JSON.

```ts
type CmsAdapterPlugin = {
  name: string;
  version: string;
  cmsProvider: string;
  syncModes: Array<"push" | "pull" | "bidirectional">;
  mappingFeatures: string[];
  conflictPolicies: string[];
  diagnosticCodes: Record<string, string>;
  pull(options: CmsPullOptions): Promise<CmsSyncResult>;
  push(page: PageBuilderPage, options: CmsPushOptions): Promise<CmsSyncResult>;
};
```

### Conformance harness

Every plugin must pass the conformance harness before being considered stable.

```ts
import { runImporterFixtureSuite } from "@pb/extensions/testkit";

const scorecard = await runImporterFixtureSuite(myImporterPlugin, fixtures);
// scorecard.score (0–100), scorecard.passed[], scorecard.failed[]
```

Tests the harness runs:

1. Schema validity: output validates against `@pb/contracts`
2. Fixture parity: output matches golden fixture within declared fidelity level
3. Diagnostics integrity: unsupported constructs produce explicit diagnostics, not silent drops
4. Version compatibility: plugin declares supported versions and behavior matches
5. Regression: same input produces same output across runs (deterministic)

### Verification

- [x] Importer, exporter, and CMS adapter type interfaces are exported from `@pb/extensions`
- [x] A reference exporter (Figma path candidate) passes the conformance harness at 100%
- [x] A plugin that silently drops an unsupported field fails the diagnostics integrity test
- [x] Harness scorecard shape is machine-readable JSON
- [x] Harness works without app internals — plain Node.js test runner

## Anti-Patterns

These are explicitly prohibited. Any PR that introduces these is blocked:

1. Importing from `apps/web` internals in any plugin or external tool
2. Treating CLI or SDK output as valid without running `validatePage`
3. Silent fallback: unsupported constructs must emit diagnostics, not be dropped silently
4. Forked schema versions with no migration story
5. Plugins coupled to private element implementation details (must use public types only)
6. Breaking `@pb/contracts` without a major version bump and migration path in `@pb/core`

---

## Minimum Integration Promise

Once a tool consumes only public surfaces and passes conformance:

1. It remains supported across compatible contract versions.
2. Breaking changes arrive with a migration guide and minimum 180-day deprecation window.
3. Diagnostic codes are stable and machine-readable — tools may key logic on them.
4. Conformance score is a public commitment: a regression in a published plugin's score
   is treated as a bug in the platform, not the plugin.
